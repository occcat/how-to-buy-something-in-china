# How to Buy Something in China - Agent SOP

本仓库用于把用户的一句购买需求转化为一份可溯源、可存档的中国电商/社媒选购报告。所有 Agent 必须用中文与用户沟通，并按下面流程执行。

## 总原则

- 先澄清，再调研。任何购买任务都必须先做一轮背景搜索和需求提问，不能直接给推荐。
- 提问必须给默认选项。若当前运行环境支持类似 Plan 模式的可选项 UI，优先用可上下左右选择的选项；否则用 Markdown 编号选项降级。
- 先保护用户体验，再采集数据。使用 `ego-browser`/Lite Ego 时必须隔离操作、静音视频、用完关闭 Tab。
- 先给证据，再给结论。参数、评价和价格都必须附来源链接；无法验证的信息标注为未确认，不得编造。
- 先扩大候选，再收敛推荐。即使用户给了具体型号，也要保留 Top 5 横向候选，用于确认是否存在更优选择。
- 先确认高成本动作，再执行。需要扫码登录、二手平台、Sub Agent 并行或长时间抓取时，先征得用户同意。

## 用户询问方式

- 询问用户时，优先使用运行环境提供的可选项交互能力，例如 Plan 模式里的选项选择 UI。
- 每个问题必须提供至少 3 个互斥且具体的默认选项，并把推荐项放在第一位，标注“推荐”。
- 选项必须说明取舍影响，不能只写“是/否/其他”。
- 永远允许用户补充自定义答案；如果没有可选项 UI，就用编号列表呈现。
- 不要一次问太多。优先问会影响候选范围和搜索路径的问题，例如预算、核心场景、二手接受度、明确雷点、登录/并行授权。

## 技能加载顺序

按阶段读取并使用 `skills/` 下的本地 Skill：

1. `skills/01-needs-analysis/SKILL.md`：需求澄清、伪需求识别、最多 3 轮问答。
2. `skills/02-env-check/SKILL.md`：macOS、Lite Ego、登录态缓存和 Sub Agent 授权检查。
3. `skills/03-ego-scraper/SKILL.md`：免打扰网页访问、社媒/电商数据采集和来源记录。
4. `skills/04-selection-tables/SKILL.md`：Top 5 候选、配置表、优缺点表、价格表。
5. `skills/05-report-export/SKILL.md`：最终 Top 3、Markdown 报告和 PDF 导出。

执行具体阶段前，读取对应 `SKILL.md`。若阶段引用脚本或参考文件，只读取本次需要的文件。

## 阶段 1：需求澄清与挖掘

目标：把用户的原始需求拆成真实需求、硬约束、软偏好和待确认的伪需求。

必须执行：

1. 先做轻量背景搜索，了解当前品类/型号的市场话术、核心卖点、常见槽点和价格区间。
2. 再向用户提问。每个问题必须给至少 3 个合适选项，优先用可选择 UI；无 UI 时用编号选项，并允许用户补充自定义答案。
3. 最多进行 3 轮搜索和问答。每轮都要减少不确定性，而不是重复泛问。
4. 如果用户输入具体型号，问题必须围绕该型号展开，例如版本、容量、渠道、是否接受二手/充新、是否担心已知缺陷。
5. 识别伪需求并向用户确认，例如“想要顶配”可能真实需求是“多开不卡”或“几年不换机”。

阶段输出必须包含：

- 用户原始需求。
- 已确认硬约束：预算、用途、不可妥协项、平台/渠道限制。
- 软偏好：品牌、外观、系统、售后、重量、性能、保值等。
- 需要验证的假设和伪需求。
- 初步候选方向和必须纳入 Top 5 的型号/品类。

## 阶段 2：环境检查与准备

目标：确认调研工具链和登录态可用。

必须执行：

1. 运行 `skills/02-env-check/check_env.sh`。
2. 如果 `uname -s` 不是 `Darwin`，告知用户 Lite Ego 当前仅支持 macOS，并结束调研流程。
3. 如果缺少 Lite Ego/`ego-browser`，指引用户安装：https://lite.ego.app。
4. 根据本次调研平台检查登录态，并缓存到 `.cache/login_status.json`。
5. 如果用户接受二手，必须检查闲鱼登录态；如果需要淘宝/京东/小红书/知乎/Bilibili 详情页，也要检查对应登录态。
6. 缺少登录态时，询问用户是否愿意手动扫码登录。
7. 询问用户是否允许启动 Sub Agent 并行搜索，并明确这会增加 Token 消耗；未获同意不得启动。

缓存和输出：

- `.cache/` 必须留在 Git 忽略列表中。
- 登录态缓存只记录站点、状态、检查时间和备注，不记录 Cookie、Token、手机号、地址等敏感信息。

## 阶段 3：候选筛选与信息收集

目标：为 Top 5 候选收集可验证证据。

候选范围：

- 泛需求：先给 5 个候选产品/型号。
- 具体型号：以该型号为核心，同时加入 4 个替代/相邻选项。
- 若用户给的需求与市场现实冲突，先说明冲突，再给可行候选。

数据来源：

- 参数：官网、品牌商城、权威数据库或专业评测参数页。
- 优点：Bilibili 评测、知乎长评、小红书种草、媒体评测。
- 缺点：评论区、知乎/小红书真实反馈、闲鱼卖家出坑理由、Bilibili 弹幕/评论。
- 价格：慢慢买、京东、淘宝/天猫、官网、闲鱼二手成交或挂价。

浏览器红线：

- 需要登录、SPA 或强反爬页面时，使用 `ego-browser`/Lite Ego。
- 打开 Bilibili 等视频网站后，立即注入静音和暂停脚本。
- 每个 Tab 提取完信息后立即关闭。
- 临时 Node.js/浏览器自动化脚本只能放在 `.cache/`，任务结束立刻删除。
- 不能打扰用户当前浏览器窗口、声音、焦点和登录状态。

证据要求：

- 每个候选至少有 1 个参数来源、1 个价格来源、2 条正向体验证据、2 条负向体验证据；无法满足时在报告中说明缺口。
- 原始链接必须保留到最终表格。

## 阶段 4：图表梳理与横纵向对比

最终 Markdown 报告必须包含 3 个表格。

### 配置表

- 表头为产品，行为参数指标。
- 每个可排序指标标注前 3：🥇、🥈、🥉。
- 排名方向要符合指标含义，例如续航越长越好、重量越轻越好、价格越低越好。
- 表格下方必须提供名词解释，用通俗语言说明每个专业参数的意义和例子。

### 优缺点表

- 重点提取用户真实体验评价。
- 同一类评价用逗号分隔，并保留原始网页超链接。
- 格式示例：`[手机发热严重](https://example.com/1), [玩游戏太烫了](https://example.com/2)`。
- 区分评测号观点、普通用户反馈和卖家出坑理由，不混在一起。

### 价格表

- 分类列出：全新、二手、神价、好价、正常价、官价。
- 二手列仅在用户接受二手或需要保值分析时启用。
- 每个价格都要关联来源链接。
- 神价/好价/正常价阈值必须基于历史价、近期成交价或平台价格区间推导，并写明依据。

## 阶段 5：报告输出与推荐

最终报告必须包含 4 个章节：

```markdown
1. 调研背景、工具与需求分析
2. 候选收集 (Top 5)
3. 选型表格
4. 最终 Top 3 推荐以及理由
```

第 1 章必须包含：

- Lite Ego/`ego-browser` 使用说明和官网链接：https://lite.ego.app。
- 用户原始需求。
- 沟通拆解后的真实需求。
- 调研范围、数据来源、登录态/二手/并行搜索授权情况。

文件输出：

- Markdown：`report/YYYY-MM-DD/产品名称-调研报告.md`。
- PDF：同目录、同名 `.pdf`。
- 日期使用当前执行环境日期；文件名去掉斜杠、冒号等不适合路径的字符。
- `report/` 必须留在 Git 忽略列表中。

推荐要求：

- 给出 Top 3，并说明每个推荐适合谁、不适合谁、购买价位和避坑点。
- 若没有任何候选满足硬约束，明确说明“不推荐购买”，并给出等待/加预算/换品类方案。

## 失败与降级

- 无法访问某平台：换可访问来源，并在报告里标注缺失。
- 登录态缺失且用户不愿登录：跳过该平台，不尝试绕过登录。
- PDF 转换工具缺失：先输出 Markdown，再说明 PDF 未生成的具体原因和可用命令。
- 数据互相矛盾：优先官网/订单页/历史价格工具；社媒观点只作为体验证据。
- 超出 3 轮需求问答仍不明确：基于已知信息给保守假设，并在报告中标注。
