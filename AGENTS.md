# How to Buy Something in China - Agent SOP

本仓库用于把用户的一句购买需求转化为一份可溯源、可存档的中国社媒平台、购买平台、比价平台选购报告。所有 Agent 必须用中文与用户沟通，并按下面流程执行。

## 总原则

- 先澄清，再调研。任何购买任务都必须先做一轮背景搜索和需求提问，不能直接给推荐。
- Plan 模式是硬性前置条件。任何购买调研任务必须在支持 Plan 模式可选项 UI 的环境中执行；如果当前环境不支持，必须在阶段 1 的第一步立即告知用户切换到 Plan 模式并退出，不得继续搜索、提问或推荐。
- 先保护用户体验，再采集数据。使用 `ego-browser`/Lite Ego 时必须隔离操作、静音视频、用完关闭 Tab。
- 先给证据，再给结论。参数、评价和价格都必须附来源链接；无法验证的信息标注为未确认，不得编造。
- 先扩大候选，再收敛推荐。任何品类都必须先建立候选长名单，再筛出所有符合条件的候选，符合条件候选最多保留 50 个；不得跳过长名单直接给固定数量推荐。
- 先确认高成本动作，再执行。需要扫码登录、二手平台、Sub Agent 并行或长时间抓取时，先征得用户同意。
- 报告产出必须隔离。没有额外要求时，调研报告产出过程不得修改 Git 管理的文件，临时文件和最终产物都必须留在被忽略的目录中。

## 用户询问方式

- 询问用户时，必须使用 Plan 模式里的选项选择 UI。
- 除 Sub Agent 并行授权外，每个问题必须提供至少 3 个互斥且具体的默认选项，并把推荐项放在第一位，标注“推荐”。
- Sub Agent 并行授权问题只能提供 2 个选项：`不并行（推荐）` / `并行`。
- 选项必须说明取舍影响，不能只写“是/否/其他”。
- 永远允许用户补充自定义答案；如果没有 Plan 模式可选项 UI，立即说明“当前对话无法提供 Plan 模式的可选择选项体验，请切换到 Plan 模式后重新发起购买调研”，然后结束流程。
- 不要一次问太多。优先问会影响候选范围和搜索路径的问题，例如预算、核心场景、二手接受度、明确雷点、登录/并行授权。

## Git 工作区与产物隔离

- 报告生成过程中的中间数据、截图、HTML、日志、临时脚本、Markdown 报告和 PDF 报告都必须写入 `.cache/` 或 `report/` 这类 Git 忽略目录。
- HTML 只允许作为 PDF 导出的临时中间格式，默认写入 `.cache/`；PDF 生成并验证完成后必须立即删除 HTML 中间文件，不得把 HTML 作为最终产物留在 `report/`。
- `.cache/` 和 `report/` 必须持续保留在 `.gitignore` 中；不得为了提交报告而移除忽略规则。
- 除非用户明确要求修改 SOP、Skill、README、脚本或其他仓库文件，普通购买调研任务不得修改 Git 管理的文件。
- 不得使用 `git add -f` 强行加入 `.cache/`、`report/` 或任何报告生成产物，除非用户明确要求版本化这些产物。
- 临时 Node.js/浏览器自动化脚本只能放在 `.cache/`，用完必须立即删除。
- 完成报告任务前必须检查 `git status --short`，确认没有报告产物、临时脚本或缓存文件进入 Git 工作区。

## 调研网页分类

所有调研网页必须按下面 3 类记录和输出，不得把评价来源、购买渠道和价格历史混在一起：

1. 社媒平台：知乎、Bilibili、小红书、论坛/社区、媒体评测和评论区等，主要用于真实体验、优缺点、口碑和避坑证据。
2. 购买平台：京东、淘宝/天猫、拼多多、闲鱼、官网/品牌商城等，主要用于当前可购买价格、库存、保修、售后、版本和官方参数。
3. 比价平台：什么值得买、慢慢买等，主要用于历史价、好价/神价阈值、价格趋势和公开补贴线索。

拼多多属于购买平台，但默认不做登录态检查和直接网页采集；只记录来自社媒平台或比价平台的公开价格线索，并标注未直接核验。

## 技能加载顺序

按阶段读取并使用 `skills/` 下的本地 Skill：

1. `skills/01-needs-analysis/SKILL.md`：需求澄清、伪需求识别、最多 3 轮问答。
2. `skills/02-env-check/SKILL.md`：macOS、Lite Ego、登录态缓存和 Sub Agent 授权检查。
3. `skills/03-ego-scraper/SKILL.md`：免打扰网页访问、社媒平台、购买平台、比价平台数据采集和来源记录。
4. `skills/04-selection-tables/SKILL.md`：候选集、配置表、优缺点表、价格表、未入选原因表。
5. `skills/05-report-export/SKILL.md`：最终 Top 3、Markdown 报告和 PDF 导出。

执行具体阶段前，读取对应 `SKILL.md`。若阶段引用脚本或参考文件，只读取本次需要的文件。

## 阶段 1：需求澄清与挖掘

目标：把用户的原始需求拆成真实需求、硬约束、软偏好和待确认的伪需求。

必须执行：

1. 首先确认当前运行环境支持 Plan 模式可选项 UI；如果不支持，立即告知用户切换到 Plan 模式并退出，不得继续执行后续步骤。
2. 先做轻量背景搜索，了解当前品类/型号的市场话术、核心卖点、常见槽点和价格区间。
3. 再向用户提问。每个问题必须给至少 3 个合适选项，必须使用 Plan 模式可选项 UI，并允许用户补充自定义答案。
4. 最多进行 3 轮搜索和问答。每轮都要减少不确定性，而不是重复泛问。
5. 如果用户输入具体型号，问题必须围绕该型号展开，例如版本、容量、渠道、是否接受二手/充新、是否担心已知缺陷。
6. 识别伪需求并向用户确认，例如“想要顶配”可能真实需求是“多开不卡”或“几年不换机”。

阶段输出必须包含：

- 用户原始需求。
- 已确认硬约束：预算、用途、不可妥协项、平台/渠道限制。
- 软偏好：品牌、外观、系统、售后、重量、性能、保值等。
- 需要验证的假设和伪需求。
- 初步候选发现方向、必须纳入长名单的型号/品类，以及需要重点覆盖的行业头部公司或品牌。

## 阶段 2：环境检查与准备

目标：确认调研工具链和登录态可用。

必须执行：

1. 运行 `skills/02-env-check/check_env.sh`。
2. 如果 `uname -s` 不是 `Darwin`，告知用户 Lite Ego 当前仅支持 macOS，并结束调研流程。
3. 如果缺少 Lite Ego/`ego-browser`，指引用户安装：https://lite.ego.app。
4. 根据本次调研需要访问的社媒平台、购买平台和比价平台检查登录态，并缓存到 `.cache/login_status.json`。
5. 如果用户接受二手，必须检查闲鱼登录态；如果需要淘宝/天猫/京东等购买平台详情页，或小红书/知乎/Bilibili 等社媒平台详情页，也要检查对应登录态。拼多多不做登录态检查，仅作为社媒平台和比价平台中的价格线索参考。
6. 缺少登录态时，询问用户是否愿意手动扫码登录。
7. 询问用户是否允许启动 Sub Agent 并行搜索，并明确这会增加 Token 消耗；选项只能是 `不并行（推荐）` 和 `并行`，未获同意不得启动。

缓存和输出：

- `.cache/` 必须留在 Git 忽略列表中。
- 登录态缓存只记录站点、状态、检查时间和备注，不记录 Cookie、Token、手机号、地址等敏感信息。

## 阶段 3：候选筛选与信息收集

目标：先建立覆盖充分的候选长名单，再为所有符合条件的候选收集可验证证据；符合条件候选最多 50 个。

候选发现与覆盖检查：

- 先调研该品类的行业头部公司/品牌、主要产品线、当前新品、前几代旗舰和仍在售的上一代高端型号。
- 泛需求：先形成候选长名单，再按硬约束筛出所有符合条件的候选；符合条件候选超过 50 个时，只保留最符合硬约束、证据最充分、购买渠道最清晰的前 50 个，并说明截断规则。
- 具体型号：以该型号为核心，同时加入同品类、同预算、同场景的替代/相邻选项；不得只在同品牌内比较，除非用户明确要求。
- 若用户给的需求与市场现实冲突，先说明冲突，再给可行候选，并把被硬约束排除的高相关产品放入未入选原因表。
- Top 候选数量不是固定目标；只有最终推荐可以收敛为 Top 3。

网页分类与数据来源：

- 社媒平台：Bilibili、知乎、小红书、论坛/社区、媒体评测和评论区，主要采集优点、缺点、长期体验、真实反馈和避坑证据。
- 购买平台：京东、淘宝/天猫、官网/品牌商城、闲鱼、拼多多等，主要采集当前价格、库存、版本、保修、售后、官方参数和购买链接。
- 比价平台：什么值得买、慢慢买等，主要采集历史价、好价/神价阈值、价格趋势和补贴线索。

- 参数：官网、品牌商城、权威数据库或专业评测参数页。
- 优点：Bilibili 评测、知乎长评、小红书种草、媒体评测。
- 缺点：评论区、知乎/小红书真实反馈、闲鱼卖家出坑理由、Bilibili 弹幕/评论。
- 价格：购买平台的当前可买价、比价平台的历史价/阈值、社媒平台的公开价格线索要分开记录；拼多多价格只通过社媒平台或比价平台公开线索作为参考，并标注未直接核验。

浏览器红线：

- 需要登录、SPA 或强反爬页面时，使用 `ego-browser`/Lite Ego。
- 打开 Bilibili 等视频网站后，立即注入静音和暂停脚本。
- 每个 Tab 提取完信息后立即关闭。
- 临时 Node.js/浏览器自动化脚本只能放在 `.cache/`，任务结束立刻删除。
- 不能打扰用户当前浏览器窗口、声音、焦点和登录状态。

证据要求：

- 每个符合条件候选至少有 1 个参数来源和 1 个价格来源；用于最终 Top 3 推荐的候选还必须尽量补足 2 条正向体验证据和 2 条负向体验证据。
- 对证据不足但仍符合硬约束的候选，不得删除；必须保留在候选集中并标注证据缺口。
- 对长名单中未入选候选集的产品，必须记录未入选原因，例如不满足预算、渠道不可验证、关键参数不符、证据不足、停售/缺货、风险过高、被同价位更优选项覆盖。
- 原始链接必须保留到最终表格。

## 阶段 4：图表梳理与横纵向对比

最终 Markdown 报告必须包含 4 个表格。

### 配置表

- 当候选数较少时，表头可以为产品、行为参数指标；当候选较多导致横向表格不可读时，改为“每行一个候选、每列一个关键指标”的纵向表格，或按价格/定位分组拆表。
- 每个可排序指标标注前 3：🥇、🥈、🥉。
- 排名方向要符合指标含义，例如续航越长越好、重量越轻越好、价格越低越好。
- 表格下方必须提供名词解释，用通俗语言说明每个专业参数的意义和例子。

### 优缺点表

- 重点提取用户真实体验评价。
- 同一类评价用逗号分隔，并保留原始网页超链接。
- 格式示例：`[手机发热严重](https://example.com/1), [玩游戏太烫了](https://example.com/2)`。
- 区分评测号观点、普通用户反馈和卖家出坑理由，不混在一起。

### 价格表

- 分类列出：全新、二手、神价、好价、正常价、官价。
- 二手列仅在用户接受二手或需要保值分析时启用。
- 每个价格都要关联来源链接。
- 神价/好价/正常价阈值必须基于历史价、近期成交价或平台价格区间推导，并写明依据。

### 未入选原因表

- 列出长名单中未进入符合条件候选集的高相关产品/型号/方案。
- 每行至少包含：产品/方案、来源线索、未入选原因、是否需要用户改变预算/场景/渠道才可重新考虑。
- 不能只写“综合不如”；必须给出可核查、可解释的排除理由。

## 阶段 5：报告输出与推荐

最终报告必须包含 4 个章节：

```markdown
1. 调研背景、工具与需求分析
2. 候选收集（最多 50 个符合条件候选）
3. 选型表格
4. 最终 Top 3 推荐以及理由
```

第 1 章必须包含：

- Lite Ego/`ego-browser` 使用说明和官网链接：https://lite.ego.app。
- 用户原始需求。
- 沟通拆解后的真实需求。
- 调研范围、社媒平台、购买平台、比价平台数据来源、登录态/二手/并行搜索授权情况。
- 行业头部公司/品牌、新品和前几代旗舰的覆盖情况，以及候选集截断规则（如果触发 50 个上限）。

文件输出：

- Markdown：`report/YYYY-MM-DD/md/产品名称-调研报告.md`。
- PDF：`report/YYYY-MM-DD/pdf/产品名称-调研报告.pdf`。
- PDF 必须经由 HTML 中间格式生成：先把最终 Markdown/报告内容渲染成带样式的临时 HTML，再由浏览器打印或 HTML-to-PDF 工具转换为 PDF。
- HTML-to-PDF 转换必须关闭默认页眉页脚；PDF 中不得出现工具自动添加的标题、URL、日期、页码等页眉页脚，除非用户明确要求。
- HTML 中间文件、临时 CSS、临时资源清单和转换日志只能放在 `.cache/`；PDF 生成并验证完成后必须删除 HTML 中间文件和仅用于转换的临时资源。
- 同一日期下必须按产物类型拆分 `md/` 和 `pdf/` 子目录，不得把 Markdown 与 PDF 混放在日期目录根部。
- 日期使用当前执行环境日期；文件名去掉斜杠、冒号等不适合路径的字符。
- `report/` 必须留在 Git 忽略列表中，不得被 `git add` 加入版本控制。
- 报告导出时只允许写入 `report/` 和 `.cache/`；没有额外要求时不得修改 Git 管理的文件。

推荐要求：

- 给出 Top 3，并说明每个推荐适合谁、不适合谁、购买价位和避坑点。
- 若没有任何候选满足硬约束，明确说明“不推荐购买”，并给出等待/加预算/换品类方案。

## 失败与降级

- 无法访问某平台：换可访问来源，并在报告里标注缺失。
- 登录态缺失且用户不愿登录：跳过该平台，不尝试绕过登录。
- 当前环境不支持 Plan 模式可选项 UI：在阶段 1 的第一步退出，不降级为 Markdown 编号选项。
- PDF 转换工具缺失：先输出 Markdown，再说明 PDF 未生成的具体原因和可用命令。
- HTML-to-PDF 链路不可用：不得退回到手写 PDF 或直接 Markdown-to-PDF 后假装完成；先输出 Markdown，并说明缺失的 HTML-to-PDF 工具或浏览器打印能力。
- 数据互相矛盾：购买平台的官网/订单页优先于社媒价格线索，比价平台的历史价工具优先用于阈值判断；社媒观点只作为体验证据。
- 超出 3 轮需求问答仍不明确：基于已知信息给保守假设，并在报告中标注。
